1. Indexing

1.1 Basic Indexing

a) Why might we break an index into shards
We break an index into shards so the data can be split across multiple nodes, which helps Elasticsearch scale and process searches faster.

b) Why might we replicate an index
Replication improves fault tolerance and read performance by keeping duplicate shard copies on other nodes.

c) Why is cluster health yellow
The cluster health is yellow because replica shards are configured, but since there is only one node in the cluster, the replicas cannot be assigned.

b) Re-run the three queries
After applying `my_analyzer` and re-indexing, results changed dramatically. 
- Query 1 ("cat") and Query 2 ("cats") now return identical results: 126 hits each with identical max scores (10.429893), confirming that stemming collapses "cat" and "cats" to the same token. 
- Query 3 ("cat cats" with AND) also returns 126 hits with max score 20.859785, because with stemming the two terms are the same stem, the AND condition is trivially satisfied for every document matching the stem, so results match Q1/Q2 exactly.

c) Did index size become larger or smaller, and why?
The index became smaller (from  31.8mb with default analyzer to 25.1mb with my_analyzer). This is due to multiple words 'cat' 'cats' 'catting' all collapsing to the same stem of 'cat', therefore the index needs ot store less unique terms.
